# 五子棋 Alpha-Beta 分析器

一套可从任意 15x15 局面开始分析的五子棋工作台。后端使用 C++20，包含
Alpha-Beta 搜索、Renju 黑方禁手判定、强制杀棋证明、启发式胜率估计和内置
HTTP/JSON 服务；前端为无外部依赖的 Canvas 棋盘编辑器。

## 功能

- 任意局面编辑、撤销、重做、清空、JSON/15 行文本导入导出
- 棋盘手数可随时显示或隐藏；导入局面的已有棋子不编号，后续新落子从 1 开始，
  从空棋盘手动摆放时则保留全部落子编号
- 显式设置当前行棋方，不根据黑白子数量推断
- 自由规则：双方五连及长连均获胜
- 连珠规则：仅黑方禁止长连、双四、双活三；白方不受禁手限制
- RIF 胜负优先级：黑方同一手形成恰好五连时优先获胜，即使另一方向同时长连
- 迭代加深 Negamax Alpha-Beta、PVS、置换表、历史与杀手启发
- 置换表优先探测并缓存精确静态叶节点和深度无关战术终局，减少重复威胁扫描
- 估价函数合并黑白扫描，胜点与候选生成使用邻域掩码和直接连线计算
- 强制威胁搜索证明 VCF 杀棋，输出总落子数和完整主变化
- 未证明杀棋时，以相邻搜索深度的中心评分输出双方互补的启发式胜率估计，抑制
  五子棋静态估值常见的奇偶层地平线摆动
- 搜索过程中逐个迭代深度刷新最佳着、胜率、节点、NPS 和主变化
- 支持无限分析并可随时停止，停止后保留最后一个完整深度的结果
- 分析过程中仍可编辑、撤销、重做、清空或导入局面；旧搜索自动取消，短暂去抖后
  串行切换到最新局面，旧任务结果不会覆盖新棋盘
- 浏览器按完整棋盘、行棋方和规则缓存最近 64 个局面的最后完整分析快照，回退或
  再次进入已计算局面时立即恢复历史静态结果
- 每个候选着都带有独立主变化，可点击查看双方后续的一路具体分支
- 支持玩家对引擎和引擎对引擎；黑白双方可分别设置初始时间和每步加秒
- 最佳着、主变化、候选着、深度、节点、剪枝、置换命中、NPS

## 构建与运行

要求 Linux 或其他提供 POSIX sockets 的系统、CMake 3.20+ 和支持 C++20 的编译器。

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build -j
ctest --test-dir build --output-on-failure
./build/gomoku_server --port 8080
```

浏览器打开 [http://127.0.0.1:8080](http://127.0.0.1:8080)。可用参数：

```text
--bind 127.0.0.1
--port 8080
--web-root ./web
```

## 性能参考

同一台测试环境、同一中盘局面和候选限制下，固定深度结果保持相同最佳着与评分：

| 规则与深度 | 优化前 | 优化后 | 总搜索节点 |
|---|---:|---:|---:|
| 自由规则，深度 5 | 1104 ms | 153 ms | 24279 -> 22917 |
| Renju，深度 4 | 202 ms | 82 ms | 10371 -> 9671 |

实际速度取决于 CPU、编译器、局面分支数和禁手复杂度。

## API

### 分析局面

`POST /api/analyze`

```json
{
  "board": [
    "...............",
    "...............",
    "...............",
    "...............",
    "...............",
    "...............",
    "...............",
    ".....BBB.......",
    "...............",
    "...............",
    "...............",
    "...............",
    "...............",
    "...............",
    "W.............."
  ],
  "sideToMove": "black",
  "rules": "freestyle",
  "options": {
    "timeLimitMs": 2000,
    "maxDepth": 5,
    "threatDepth": 9,
    "candidateLimit": 16
  }
}
```

坐标以左上角为 `(0, 0)`，`x` 向右、`y` 向下。棋盘字符支持 `B/X/1`、
`W/O/2` 和 `./_/-/0`。

已证明杀棋时：

```json
{
  "kind": "forced_mate",
  "proven": true,
  "winner": "black",
  "mateIn": 3,
  "bestMove": { "x": 8, "y": 7 },
  "pv": [
    { "x": 8, "y": 7 },
    { "x": 4, "y": 7 },
    { "x": 9, "y": 7 }
  ]
}
```

`mateIn` 是从当前局面开始、双方合计的总落子数。普通分析返回
`kind: "evaluation"`、`proven: false`、双方胜率、候选着和统计信息。
已证明和棋返回 `drawRate: 1`，双方胜率均为 0。

同步接口只用于有限分析。`options.infinite: true` 必须使用下面的异步接口。
每个候选项除 `move`、原始 `score`、配对后的 `winRateScore` 和 `winRate` 外，
还包含从该候选着开始的 `pv`。

### 实时与无限分析

启动后台分析：

`POST /api/analyze/start`

请求体与 `/api/analyze` 相同。有限分析使用常规时间和深度限制；无限分析设置：

```json
{
  "options": {
    "timeLimitMs": 2000,
    "maxDepth": 8,
    "threatDepth": 9,
    "candidateLimit": 16,
    "infinite": true
  }
}
```

返回 `202`：

```json
{ "jobId": 2, "running": true, "version": 0 }
```

轮询新结果：

`POST /api/analyze/poll`

```json
{ "jobId": 2, "version": 3 }
```

`version` 传客户端已收到的版本。每完成一个迭代深度，返回值中的 `version`
递增，`changed: true` 且 `result` 包含该完整深度的快照；没有新结果时
`changed: false`。`running: false` 表示任务已经完成或停止。

停止分析：

`POST /api/analyze/stop`

```json
{ "jobId": 2 }
```

服务最多同时执行 4 个后台分析任务。无限任务超过 30 秒未被轮询会自动取消；
已完成的任务结果保留约 5 分钟。

### 对弈

网页中的“对弈”模式可从当前局面开始玩家对玩家、玩家对引擎或引擎对引擎。
黑白双方各自使用 Fischer 时钟，可独立设置初始分钟数和每步增益秒数。暂停会
冻结当前时钟并停止正在进行的引擎搜索，继续后由当前方重新计算。

### 检查落点

`POST /api/forbidden` 使用同一局面字段，另加：

```json
{ "move": { "x": 7, "y": 7 } }
```

返回 `legal`、`forbidden`、`type`、`reason` 和 `winning`。`type` 可能为
`overline`、`double_four`、`double_three`、`occupied` 或 `none`。

### 健康检查

`GET /api/health`

## 结果语义

- Alpha-Beta 分数不是统计概率。`score` 保留当前完整深度的原始搜索分数；普通
  非终局分析中的 `blackWinRate` 和 `whiteWinRate` 由当前与上一搜索深度的中心
  分数 `winRateScore` 经过对称 logistic 映射得到，二者之和为 1。深度 1 与搜索前
  的静态深度 0 评分配对，避免评估条因地平线最后一手属于不同方而按奇偶层跳变。
- 只有 `proven: true` 的 `forced_mate` 才显示“几步杀”。当前精确证明覆盖立即胜、
  双重一步胜威胁和连续成四威胁的 VCF；普通选择性 Alpha-Beta 搜索即使得到极高
  分数，也不会被包装成已证明杀棋。
- 任意棋盘快照无法还原历史落子是否合法。Renju 模式只检查分析开始后的黑方候选
  着法；输入局面不因棋子数量不平衡而拒绝，但双方同时已有胜线会返回 422。
- 同步分析请求与后台分析任务都有限流。同步请求断开后，取消信号会在下一个搜索
  节点终止计算；后台任务通过停止接口或无限分析的轮询租约释放资源。
